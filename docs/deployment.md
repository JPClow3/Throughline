# Deployment (Docker + Dokploy on EC2)

This guide deploys the app as a Docker Compose stack managed by
[Dokploy](https://dokploy.com) (a self-hosted PaaS that uses Traefik for ingress
and Let's Encrypt TLS) on an AWS EC2 instance.

## Architecture / connection map

```
                         ┌────────────────────────── EC2 host ──────────────────────────┐
   Browser  ──HTTPS──▶  Traefik (Dokploy)                                                │
   (PWA)               │   ├─ Host(APP_HOST) /        ─────────▶  web (nginx :80)        │
                       │   └─ Host(APP_HOST) /api/*   ──strip──▶  push-api (Fastify :8787)│
                       │                                              │  ▲                │
   Browser SW  ◀──Web Push (FCM/Mozilla/Apple)◀── web-push (VAPID) ───┘  │ POST /dispatch-due
                       │                                          push-store.json (volume) │
                       │   ofelia (cron) ──docker exec every 5m──────────┘                │
                       └───────────────────────────────────────────────────────────────┘
```

Hops:

1. **Browser → web**: Traefik terminates TLS and serves the static PWA (nginx, SPA fallback to `/index.html`).
2. **Service worker**: precaches the shell and receives Web Push `push` events. Generated by `vite-plugin-pwa`.
3. **Web → push-api**: same-origin `fetch` to `/api/*` (`VITE_PUSH_API_URL=/api`). Traefik strips `/api` and forwards to `push-api:8787`. Endpoints: `POST /subscriptions`, `PUT /subscriptions/:hash/reminders`, `DELETE /subscriptions/:hash`, `POST /reminders` (legacy), `GET /health`.
4. **push-api → push services**: `web-push` signs payloads with the `VAPID_*` keys.
5. **ofelia → push-api**: `docker exec` runs `POST /dispatch-due` every 5 min (internal; protected by `DISPATCH_TOKEN`).
6. **push-api → disk**: reads/writes `push-store.json` on the `push-data` volume (single-instance; not horizontally scalable).

## Prerequisites

- A domain (or subdomain) you can point at the EC2 box. **Required** — Web Push and installable PWAs only work over HTTPS.
- VAPID keys: `npm run vapid -w apps/push-api` → copy the public/private keys.

## 1. Provision the EC2 instance

- AMI: Ubuntu 22.04/24.04 LTS. Size: **t3.medium** recommended (images are built on the host; a t3.small works only with swap added).
- Storage: 20 GB+.
- **Security group** inbound: `22` (SSH, your IP), `80` (HTTP, anywhere), `443` (HTTPS, anywhere).
- Allocate an Elastic IP and point your domain's `A` record (`APP_HOST`) at it.

## 2. Install Docker + Dokploy

SSH in and run the Dokploy installer (installs Docker + Traefik):

```bash
curl -sSL https://dokploy.com/install.sh | sh
```

Open `http://<elastic-ip>:3000`, create the admin account.

## 3. Create the Compose application

1. In Dokploy: **Create Project → Create Service → Compose**.
2. Point it at this repo (Git) and set the compose path to `docker-compose.yml`.
3. **Environment** — paste from your filled `.env` (see `.env.example`):
   - `APP_HOST` = your domain
   - `VAPID_PUBLIC_KEY`, `VAPID_PRIVATE_KEY`, `VAPID_SUBJECT`
   - `VITE_VAPID_PUBLIC_KEY` = same as `VAPID_PUBLIC_KEY` (baked into the web bundle)
   - `SESSION_SECRET` = a long random string (signs account session cookies)
   - `DISPATCH_TOKEN` = a long random string (protects `/dispatch-due`)
   - `CORS_ORIGIN` = `https://<APP_HOST>` (optional; tightens CORS)

Accounts + encrypted sync use SQLite at `/data/throughline.db` on the same `push-data`
volume; content is end-to-end encrypted on-device, so the server stores only ciphertext.
4. The compose file already attaches services to `dokploy-network` and adds the
   Traefik routers/TLS labels. If your Dokploy Traefik uses a non-default cert
   resolver/entrypoint, override `TRAEFIK_CERTRESOLVER` / `TRAEFIK_ENTRYPOINT`.
5. **Deploy**. Dokploy builds both images and starts web + push-api + ofelia.

The `push-data` volume persists the push store across redeploys/restarts.

## 4. Verify

```bash
curl https://<APP_HOST>/api/health
# => {"ok":true,"vapidConfigured":true,"subscriptions":0,"reminders":0}
```

Then in a browser at `https://<APP_HOST>`:

1. Open **Settings → enable notifications → Subscribe** (`POST /api/subscriptions` returns an `endpointHash`).
2. **Sync** (`PUT /api/subscriptions/:hash/reminders`).
3. Create a task with a near-future reminder time; within ~5 min ofelia fires
   `/dispatch-due` and the device receives a push notification.
4. Confirm `https://<APP_HOST>/api/dispatch-due` (POST) without the bearer token returns `401`.

## Redeploys & data

- Push **to the tracked branch** → redeploy from Dokploy (or enable auto-deploy webhooks).
- The JSON store lives on the `push-data` volume; back it up if you care about subscriptions.

## Troubleshooting

- **`vapidConfigured:false`** → VAPID env vars not set on `push-api`.
- **404 on `/api/...`** → Traefik StripPrefix/priority labels not applied, or the API container is unhealthy (`GET /health`).
- **No push received** → must be HTTPS with a valid cert; check the browser granted notification permission and that ofelia is running (`docker logs <ofelia>`).
- **Build OOM on a small instance** → add swap or use t3.medium, or build images in CI and push to a registry (see below).

## Scaling notes (future)

- The push store is a single JSON file on a volume → run **one** push-api replica only. Moving to SQLite/Redis is required before scaling out.
- For lighter EC2 boxes, build images in CI (GitHub Actions) and pull from a registry instead of building on the host.
